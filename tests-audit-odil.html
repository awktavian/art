🔺 Why The Pyramid Looks "Off"

Q: The classic testing pyramid is Unit (70%) → Integration (20%) → E2E (10%). This codebase is Unit (24%) → Integration (61%) → E2E (2%). What's going on?

Short answer: This is a domain-specific inversion, not bad practice.

E2E 218 tests (2%)
Integration 6,360 tests (61%)
Unit 2,554 tests (24%)

Why this makes sense for Kagami:

  • Safety-critical system: CBF (Control Barrier Functions) require multi-component verification. A safety invariant like h(x) ≥ 0 can't be tested in isolation — it needs the full constraint solver, world model, and actuator chain.
  • ML/AI codebase: Testing a neural network's forward pass in isolation (unit test) proves nothing. You need integration tests that verify the model works with real data pipelines.
  • Colony architecture: 7 colonies (Spark, Forge, Flow, etc.) are designed to work together via Fano plane routing. Unit tests for individual colonies miss the emergent behavior.

The F1 racing analogy was apt: You don't just test the engine (unit) — you test the engine + chassis + aerodynamics together (integration) because that's where race performance lives.

Q: But doesn't this make the test suite slow?

Yes, and that's the tradeoff.

Classic Pyramid (70/20/10)

Unit tests <1s each
Full suite ~15-30 min
Feedback loop Fast

Kagami's Pyramid (24/61/2)

Integration tests <5s each
Full suite ~8-9 hours
Feedback loop Tiered (CI strategy)

The solution is tiered CI: run unit tests on every commit (45 min), integration tests on PR merge (nightly), E2E tests weekly.
